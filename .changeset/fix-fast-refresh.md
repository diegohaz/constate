Fix React Fast Refresh discarding state and dropping updates when the file calling `constate(...)` is edited

In development, `constate(useValue, ...)` now reuses the previously returned `Provider` and hooks across module re-evaluations when the call comes from the same source location and the structural signature of `useValue` and selectors is unchanged. The cache key combines the caller's `file:line:column` (captured from `new Error().stack` after skipping constate's own frames — first by function name match against `getCallerLocation` / `getCacheKey` / `constate`, then by file path under known dependency-resolver roots `node_modules`, `.yarn`, and `.vite` covering published `constate/dist/index.js`, pre-bundled `node_modules/.vite/deps/constate*.js`, and yarn-pnp `.zip` layouts) with `useValue.name`, the selector count, the selector names, and the hook counts of `useValue` and each selector. The cached `Provider` reads through internal refs (`useValueRef.current`, `selectorsRef.current[i]`) so edits to the hook body or selector bodies take effect on the next render while React preserves the Provider's fiber and its hook state. Adding or removing hook calls inside `useValue` or any selector invalidates the cache entry and falls back to creating a fresh Provider — preventing React's "rendered more hooks" error at the cost of state loss for that one structural edit. Two `constate(...)` calls on distinct source locations (different line, or same line but different column) stay independent, anonymous `useValue` functions and stack formats the parser can't recognize bypass the cache, and production builds (`process.env.NODE_ENV === "production"`) skip the cache entirely. Runtime overhead in production is `1 + selectorCount` ref dereferences per Provider render. Known limitations of the dev-only cache: the hook-count heuristic is a regex over `fn.toString()` that matches `\buse[A-Z]\w*\s*\(`, so it does not see React 19's `React.use(...)` (no capital after `use`), hooks aliased through other identifiers, or same-count reorderings (e.g. swapping one `useMemo` for one `useCallback`); two `constate(...)` calls reached through the same line and column of a wrapper helper that itself calls `constate` will be treated as the same call site; edits above the `constate(...)` call that shift its line, or LHS edits that shift its column, invalidate the cache for that call site; and stack-frame parsing supports V8, JSC, and SpiderMonkey but cannot promise correctness for synthetic runner frames such as a webpack bootstrap shim.
